# 429 Too Many Requests 재시도 설계

## 배경

호요버스 출석체크 API 호출 시 간헐적으로 HTTP 429 (Too Many Requests) 에러가 발생하고 있다. 현재 에러 발생 시 `lastCheckInDate`가 갱신되어 다음 일일 리셋(UTC 16:00)까지 재시도하지 않는 구조이다.

## 목표

429 에러 발생 시 `lastCheckInDate`를 갱신하지 않아, 다음 자동 출첵 알람(30분 후)에서 자연스럽게 재시도되도록 한다.

## 범위

- 429 에러만 재시도 대상. 다른 에러(`AuthExpired`, `NotLoggedIn`, `NeedCaptcha` 등)는 현행 유지.
- API 호출 간 딜레이 추가는 범위 밖.
- 동시 실행 방지(mutex)는 범위 밖.

## 설계

### 1. HTTP 레이어 — 429 감지

`src/shared/errors/TooManyRequestsError.ts`에 전용 에러 클래스를 추가한다. 기존 `JsonParseError`, `RetryLaterError`와 동일한 패턴을 따른다. `RetryLaterError`는 API retcode 기반 에러(scrap 서비스)에 사용되므로 HTTP 상태 코드 기반인 429는 별도 클래스로 분리한다.

```typescript
export class TooManyRequestsError extends Error {
  constructor(public readonly url: string) {
    super(`Too Many Requests: ${url}`);
    this.name = 'TooManyRequestsError';
  }
}
```

`src/shared/utils/http.ts`의 `parseJsonResponse`에서 JSON 파싱 전에 HTTP 상태 코드를 확인한다.

```
parseJsonResponse(res, url)
  if res.status === 429 → throw TooManyRequestsError(url)
  else → 기존 JSON 파싱 로직
```

`httpWithCookie`의 catch 블록에서 `TooManyRequestsError`는 `JsonParseError`와 마찬가지로 `captureApiException` (Sentry 보고)을 **건너뛴다**. 429는 일시적 에러이므로 Sentry에 보고할 필요가 없다.

### 2. API retcode 상수 추가

`src/shared/constants/api-ret-code.ts`의 `ApiRetCode` enum에 추가한다. 이 값은 Hoyolab API retcode가 아닌 HTTP 상태 코드이지만, `CallCheckInApiOutput.retCode`가 `ApiRetCode` 타입으로 정의되어 있어 타입 호환을 위해 동일 enum에 포함한다.

```typescript
// HTTP status code (Hoyolab API retcode 아님)
TooManyRequests = 429,
```

`ErrorMessageKey`에도 해당 항목을 추가한다.

```typescript
[ApiRetCode.TooManyRequests]: 'common.too_many_requests',
```

### 3. i18n 메시지 추가

`src/shared/i18n/translations.ts`에 429 에러 메시지를 추가한다.

- ko: `'요청이 너무 많습니다. 다음 출석체크 때 재시도합니다.'`
- en: `'Too many requests. Will retry at next check-in.'`
- ja / zh_TW: 각 언어에 맞는 번역

### 4. 비즈니스 레이어 — CallCheckInApiService

`src/apps/background/service/check-in/CallCheckInApiService.ts`의 `execute` 메서드에서 `httpWithCookie` 호출을 try-catch로 감싼다.

- `TooManyRequestsError` catch 시: `{ actId, ltuid, retCode: ApiRetCode.TooManyRequests, lastCheckInMessage: i18n.t('common.too_many_requests') }` 형태로 결과 목록에 push.
- 루프는 중단하지 않고 다음 계정으로 계속 진행한다.
- `TooManyRequestsError`가 아닌 예외는 기존처럼 throw하여 상위로 전파한다.

### 5. 스토어 레이어 — lastCheckInDate 갱신 조건 분기

`src/apps/background/store/accountStore.ts`의 `updateLastCheckIn`에서 `retCode`가 `TooManyRequests`일 때:

- `lastCheckInMessage`, `retCode`, `lastCheckInResult`는 갱신한다 (`lastCheckInResult`는 기존 default 분기에 의해 `'fail'`).
- `lastCheckInDate`는 **갱신하지 않는다**.

이렇게 하면 다음 30분 알람 시 `GetCheckInListService`의 필터(`lastCheckInDate < getLastCheckInAvailDate()`)를 통과하여 재시도된다.

### UI 동작

429 발생 시 UI에는 기존 에러와 동일하게 `'fail'` 상태로 표시되며, 에러 메시지가 표시된다. 별도의 "재시도 중" UI는 추가하지 않는다. 30분 후 자동 재시도되어 성공하면 정상 상태로 변경된다.

## 변경 파일

| 파일 | 변경 내용 |
|------|----------|
| `src/shared/errors/TooManyRequestsError.ts` | 신규 — 에러 클래스 |
| `src/shared/utils/http.ts` | `parseJsonResponse`에서 429 감지, `httpWithCookie`에서 Sentry 제외 |
| `src/shared/constants/api-ret-code.ts` | `TooManyRequests = 429` 및 `ErrorMessageKey` 추가 |
| `src/shared/i18n/translations.ts` | 429 에러 메시지 추가 (ko, en, ja, zh_TW) |
| `src/apps/background/service/check-in/CallCheckInApiService.ts` | try-catch로 `TooManyRequestsError` 처리 |
| `src/apps/background/store/accountStore.ts` | 429일 때 `lastCheckInDate` 갱신 안 함 |

## 데이터 흐름

```
알람 발동 → checkInAll()
  → GetCheckInListService.execute()
    → lastCheckInDate 미갱신 계정 필터링 (429였던 계정 포함)
  → CallCheckInApiService.execute(targets)
    → httpWithCookie(url, ...) → fetch
      → 429 응답 → TooManyRequestsError throw
    → catch TooManyRequestsError → 결과 목록에 retCode: 429로 기록, 루프 계속
  → accountStore.updateLastCheckIn(results)
    → retCode === 429 → lastCheckInDate 유지 (다음 알람에서 재시도)
    → retCode !== 429 → lastCheckInDate 갱신 (기존 동작)
```

## 알려진 제한사항

- 429가 지속적으로 발생하면 30분마다 무한히 재시도한다. 재시도 횟수 제한이나 백오프는 이번 범위에 포함하지 않으며, 필요시 후속 작업으로 진행한다.
